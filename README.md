# Weekly Paper: A/B 테스트 및 이벤트 로그 설계

## 1. A/B 테스트의 장단점 및 해결 방안

### 장점
- 데이터 기반의 의사결정: 직관이 아닌 실제 유저 데이터로 승자를 판별함.
- 리스크 방지: 소수 유저에게 먼저 검증하여 대규모 장애나 매출 하락 위험을 낮춤.
- 인과관계 확인: 특정 기능의 변경이 지표 변화에 미친 영향을 명확히 파악 가능함.

### 단점 및 해결 방안
- 시간 및 비용 소모: 유의미한 표본을 모으는 데 물리적인 시간이 필요함.
  - 해결: 트래픽이 높은 페이지를 우선 테스트하거나 MAB 알고리즘을 도입함.
- 새로움 효과: 단지 새로운 기능이라서 일시적으로 클릭률이 오르는 현상이 발생함.
  - 해결: 테스트 기간을 최소 1~2주 이상 유지하여 데이터를 안정화함.
- 실험 설계 편향: 잘못된 표본 할당으로 결과가 왜곡될 수 있음.
  - 해결: SRM 검사를 통해 실험군과 대조군의 할당 비율을 실시간으로 모니터링함.

## 2. 유의미한 결과가 없을 때의 해석과 다음 단계

성과 차이가 미미하거나 p-value가 기준치를 넘지 못하는 상황에 대한 대처법임.

### 결과 해석
- 실질적 차이 없음: 두 버전 간의 유저 경험 차이가 실제로는 존재하지 않는 경우임.
- 표본 부족: 실제로는 차이가 있으나 유저 수가 적어 통계적으로 증명하지 못한 상태임.
- 지표 상충: 일부 지표는 올랐으나 다른 지표가 하락하여 전체 합이 0에 수렴한 경우임.

### 다음 단계
- 세그먼트 분석: 신규 유저나 특정 유입 경로별로 데이터를 쪼개서 숨은 성과를 찾아냄.
- 가설 재검토: 변경 사항이 유저에게 큰 임팩트를 주지 못했는지 확인하고 더 과감한 변화를 시도함.
- 정성적 조사: 유저 인터뷰나 사용성 테스트를 통해 수치 뒤에 숨은 심리적 이유를 파악함.

## 3. 이벤트 데이터 로그 설계 (Event Taxonomy)

### 주요 구성 요소 및 역할

| 구성 요소 | 역할 | 예시 |
| :--- | :--- | :--- |
| Event Name | 유저가 수행한 구체적인 행동을 정의함 | click_add_to_cart, login |
| Properties | 이벤트가 발생한 시점의 상세한 맥락 정보임 | product_id, item_price |
| User Properties | 이벤트를 발생시킨 유저의 고유한 특성임 | user_level, gender |
| Common Properties | 모든 이벤트에 공통으로 들어가는 환경 정보임 | os_type, app_version, timestamp |

### 설계 원칙
- 명명 규칙 통일: Object_Action 형식을 사용하는 등 팀 내 규칙을 정해 가독성을 높임.
- 플랫폼 간 일관성: 웹과 앱 등 서로 다른 환경에서도 동일 행동은 같은 이름으로 남김.
- 관리 효율성: 서비스 확장 시 속성을 추가하기 쉬운 구조로 설계함.
